# Frontend Update Guide — v1.1 New Features

This file explains every new field the backend `/upload` endpoint now returns,
what it looks like, and exactly what to build in the UI for each one.

---

## What changed in the API response

The `/upload` endpoint returns 4 new top-level fields alongside the existing
`results`, `totals`, and `assumptions`.

---

## 1. `warnings` — Anomaly Flags

### What it is
An array of data-quality or methodological issues detected in the triangle.
Can be empty `[]` if no anomalies were found.

### Example JSON
```json
"warnings": [
  {
    "type": "bf_cl_divergence",
    "severity": "warn",
    "affected_years": [2022, 2023],
    "message": "BF and Chain Ladder IBNR diverge by more than 15% for accident year(s): 2022, 2023. Large divergence typically signals immature years..."
  },
  {
    "type": "concentration_risk",
    "severity": "warn",
    "affected_years": [2021],
    "message": "High loss concentration detected: AY 2021 (44.2%) of total paid losses..."
  }
]
```

### Possible `type` values
| type | severity | Meaning |
|---|---|---|
| `ldf_below_unity` | error | Paid losses decreased between periods — likely a data error |
| `concentration_risk` | warn | One accident year is >40% of total paid losses |
| `insufficient_periods` | warn | Fewer than 3 credible development periods |
| `tail_double_count` | warn | Tail factor and final LDF are both elevated (risk of overcounting) |
| `bf_cl_divergence` | warn | BF and CL IBNR differ by >15% on one or more years |

### What to build
- If `warnings` is empty: show nothing (or a subtle green "No anomalies detected")
- If `warnings` has items: show a **Warnings panel** above the results table
- Each warning card should show:
  - An icon: yellow triangle for `warn`, red circle for `error`
  - The `message` text
  - If `affected_years` is not empty: show the years as small badges
- Collapse the panel by default if all warnings are `warn` severity
- Always show expanded if any warning is `error` severity

## 2. `recommended_method` + `recommendation_rationale`

### What it is
The engine's recommendation for which of the three methods (CL, BF, or CC) is
most appropriate for this specific triangle, plus a plain-English explanation
of why.

### Example JSON
```json
"recommended_method": "CC",
"recommendation_rationale": "The average CDF to ultimate is 1.43, indicating an immature portfolio (3 of 6 accident years have fewer than 48 months of development). Premium data is available, allowing Cape Cod to derive a data-driven ELR from used-up premium. CC is preferred over BF when premiums are credible because the ELR is endogenously derived rather than externally assumed."
```

### Possible values for `recommended_method`
`"CL"` | `"BF"` | `"CC"`

### What to build
- Show a **Method Recommendation** card/banner near the top of the results page
- Highlight the recommended method column in the results table (e.g. a subtle
  blue background or a "Recommended" badge on the column header)
- Show the `recommendation_rationale` text below the badge — this is already
  plain English, no reformatting needed
- Example card design:
  ```
  ┌─────────────────────────────────────────────────────┐
  │  ✦ Recommended Method: Cape Cod                     │
  │                                                     │
  │  The average CDF to ultimate is 1.43, indicating    │
  │  an immature portfolio...                           │
  └─────────────────────────────────────────────────────┘
  ```

## 3. `commentary` — AI Narrative

### What it is
A plain-English paragraph (≤300 words) generated by Claude that explains:
- Which method produced the highest/lowest IBNR and by how much
- Which accident years are most immature and therefore most sensitive
- Whether the CC ELR differs materially from the user-supplied ELR
- Any notable BF-CL divergences

**This field can be `null`** if the `ANTHROPIC_API_KEY` is not set on the server.
The UI must handle `null` gracefully (just hide the section).

### Example JSON
```json
"commentary": "The three methods produced materially different total IBNR estimates: Cape Cod ($4.2M) was the highest, followed by Bornhuetter-Ferguson ($3.9M) and Chain Ladder ($3.4M). The $800K spread between CC and CL is driven primarily by accident years 2022 and 2023, which have only 24 and 12 months of development respectively — making their Chain Ladder projections highly sensitive to early volatility.\n\nAccident year 2023 (12 months developed) carries the most uncertainty; a 10% shift in the assumed tail factor would move its IBNR estimate by approximately $180K. The Cape Cod derived ELR of 61.3% is slightly below the user-supplied BF ELR of 65%, implying the emerging experience is modestly more favorable than the a priori expectation.\n\nKey Risk: BF and CL IBNR diverge by 22% on accident year 2023, indicating this year's reserve estimate is highly sensitive to the ELR assumption."
```

### What to build
- Show an **AI Commentary** section below the results table
- Use a distinct visual treatment — e.g. a light blue/purple card with a small
  sparkle or AI icon to distinguish it from the data tables
- Render the text as-is (it's already formatted for readability)
- If the last sentence starts with `"Key Risk:"` — pull it out and render it
  as a highlighted warning chip/callout in amber/orange
- If `commentary` is `null`: hide the section entirely (don't show "Loading..." —
  just omit it silently)
- Add a small disclaimer below the commentary:
  `"AI commentary is for decision support only and does not constitute an actuarial opinion."`

## 4. `key_risk_flag` — Risk Alert

### What it is
A boolean (`true` / `false`) that is `true` when BF and CL IBNR diverge by
more than 15% on any accident year. This is computed from the data directly
(not from the AI) so it is always present even if `commentary` is `null`.

### Example JSON
```json
"key_risk_flag": true
```

### What to build
- If `true`: show a **Key Risk** badge or banner in amber/orange near the top
  of the results section (near the totals row or the method recommendation card)
  - Text: `"Key Risk: BF and CL estimates diverge significantly on one or more accident years. Review ELR assumption."`
- If `false`: show nothing
- This is intentionally redundant with the `bf_cl_divergence` warning — the
  flag is a prominent top-level signal; the warning gives the detail

## Putting it all together — suggested page layout

```
┌──────────────────────────────────────────────────────┐
│  [Method Recommendation card]                        │
│  "Recommended: Cape Cod"  + rationale text           │
├──────────────────────────────────────────────────────┤
│  [Key Risk banner — amber, only if key_risk_flag]    │
├──────────────────────────────────────────────────────┤
│  [Warnings panel — only if warnings.length > 0]      │
│  ⚠ Warning 1   ⚠ Warning 2                          │
├──────────────────────────────────────────────────────┤
│  [Results table — existing]                          │
│  Columns: AY | Paid | CL | BF | CC* (highlighted)    │
│                                        *recommended  │
├──────────────────────────────────────────────────────┤
│  [AI Commentary card — only if commentary != null]   │
│  ✦ AI Commentary                                     │
│  <commentary text>                                   │
│  [Key Risk: ... callout if present in text]          │
│  _disclaimer_                                        │
└──────────────────────────────────────────────────────┘
```
